# Internationalization (i18n)

OpenRig ships in **pt-BR** and **en-US**. The default UI language is
**en-US** — when the OS locale is not Portuguese or English, the app
falls back to English (not to the Portuguese source language). Portuguese
variants (pt-PT, pt-AO, etc.) route to pt-BR because the Portuguese
translation is more useful to a Portuguese reader than English.

**Translation keys are symbolic identifiers in English**, not source-
language strings. Every locale (including pt-BR) translates from the
same key. This is the standard convention for any project that ships
in multiple languages — a translator never sees Portuguese inside the
source code.

Examples:
```slint
LauncherButton { label: @tr("btn-new-project"); }
Text { text: @tr("title-new-project"); }
TextInput { placeholder-text: @tr("placeholder-search-projects"); }
```

```yaml
# crates/adapter-gui/locales/pt-BR.yml
btn-new-project: "Novo projeto"
title-new-project: "Novo Projeto"
placeholder-search-projects: "Buscar projetos"

# crates/adapter-gui/locales/en-US.yml
btn-new-project: "New project"
title-new-project: "New Project"
placeholder-search-projects: "Search projects"
```

A language selector at the top-right of the desktop UI lets the user
override OS detection. Picking a language persists to `config.yaml`
and takes effect on next launch.

## Key naming convention

Keys are kebab-case with a context prefix. The prefix communicates *what*
the string is so a translator can localize tone (e.g. buttons can be
imperative, labels descriptive).

| Prefix | Use for | Examples |
|---|---|---|
| `btn-` | Button labels | `btn-cancel`, `btn-new-project`, `btn-save` |
| `label-` | Form/section labels | `label-project-name`, `label-instrument` |
| `title-` | Headings, page titles, window titles | `title-new-project`, `title-spectrum` |
| `tooltip-` | Hover tooltips | `tooltip-open-project`, `tooltip-save` |
| `placeholder-` | Input placeholder text | `placeholder-search-projects` |
| `accessible-` | Accessible labels for screen readers | `accessible-language-selector` |
| `msg-` | User-facing messages, toasts, errors | `msg-no-project-loaded`, `msg-saved` |
| `dialog-` | Dialog content text | `dialog-confirm-delete-block` |
| `empty-` | Empty-state messages | `empty-no-active-inputs` |

## Two catalogs, one selector

OpenRig has **two parallel translation catalogs** that share the same
locale code:

| Side | Marker | Catalog | Format |
|---|---|---|---|
| Slint UI | `@tr("...")` | `crates/adapter-gui/translations/<lang>/adapter-gui.po` | gettext `.po` → `.mo` |
| Rust code | `rust_i18n::t!("...")` | `crates/adapter-gui/locales/<lang>.yml` | rust-i18n YAML |

This split exists because Slint's `@tr(...)` macro only speaks gettext.
For Rust we picked **rust-i18n** (most popular Rust i18n framework — 12k+
downloads/day): it's idiomatic, integrates as a compile-time macro, and
needs no external CLI tools at build time.

The single language selector in the UI sets both catalogs' active locale
on startup via `i18n::init_translations`.

## Architecture

| Layer | Tool / Mechanism |
|---|---|
| String marking (Slint) | `@tr("...")` |
| String marking (Rust) | `rust_i18n::t!("...")` |
| Extraction (Slint) | `slint-tr-extractor` (POT generation) |
| Extraction (Rust) | `cargo i18n` or `scripts/migrate-rust-strings.py --dry-run` |
| Per-locale catalogs (Slint) | gettext `.po` files |
| Per-locale catalogs (Rust) | rust-i18n YAML |
| Compilation (Slint) | `msgfmt` invoked from `crates/adapter-gui/build.rs` |
| Runtime resolution (Slint) | `gettext-rs` + Slint feature `gettext` |
| Runtime resolution (Rust) | `rust-i18n` macro + `rust_i18n::set_locale` |
| Locale detection | `sys-locale` crate (override via `config.yaml`) |
| Selector UI | `crates/adapter-gui/ui/components/language_selector.slint` |
| Selector wiring | `crates/adapter-gui/src/language_wiring.rs` |
| Locale init | `crates/adapter-gui/src/i18n.rs` (called from `main.rs`) |

The gettext text domain is **`adapter-gui`** (matches Slint's default,
which uses `CARGO_PKG_NAME`). End users never see this identifier.

## Adding or modifying translatable strings

Mark every user-visible string with `@tr(...)`. Examples:

```slint
Text { text: @tr("Save Project"); }
Text { text: @tr("Hello, {}", name); }      // Placeholders OK
Text { placeholder-text: @tr("Search…"); }
```

Do **not** concatenate translatable parts (`@tr("Hello, ") + name`) — it
breaks word ordering for languages where the noun comes first.

After adding strings, run:

```bash
./scripts/extract-translations.sh
```

This regenerates `translations/adapter-gui.pot` and merges new entries
into every per-locale `adapter-gui.po`. Existing translations are
preserved by `msgmerge`.

Then translate the new `msgstr ""` entries in each
`translations/<lang>/adapter-gui.po`. Build (`cargo build -p adapter-gui`)
to compile them to `.mo` and pick them up at runtime.

## Adding a new language

1. Pick a BCP 47 code matching what the OS reports
   (e.g. `es-419`, `fr-FR`).
2. Add an entry to `crates/adapter-gui/src/i18n.rs::SUPPORTED_LANGUAGES`.
3. Add the language code mapping to `i18n.rs::normalize`.
4. Create `crates/adapter-gui/translations/<lang>/adapter-gui.po` (you
   can copy from `pt_BR/adapter-gui.po` as a starting point). Use
   **POSIX format** for the directory name (`pt_BR`, `en_US`, `de_DE`,
   underscore not hyphen) — gettext on Unix derives the path from
   `setlocale(LC_MESSAGES, ...)` which expects underscore. The `code`
   exposed in the language selector and `config.yaml` stays in
   BCP 47 hyphen format (`pt-BR`); `init_translations` converts hyphen
   to underscore before calling `setlocale`.
5. Run `./scripts/extract-translations.sh` to fill missing entries.
6. Translate.
7. Build and ship.

## How the runtime finds translations

Search order in `i18n::resolve_translations_dir`:

1. `OPENRIG_TRANSLATIONS_DIR` env var (developer override)
2. `<exec_dir>/translations` (Windows next-to-exe, macOS `.app/Resources`)
3. `<exec_dir>/../share/openrig/translations` (Linux FHS / Debian package)
4. `<exec_dir>/../Resources/translations` (macOS `.app` fallback)
5. `CARGO_MANIFEST_DIR/translations` (debug builds via `cargo run`)

If none exist, gettext returns the source string (passthrough), which is
correct for the source language (pt-BR).

## Build host requirements

`gettext-rs` uses the system `libintl` and the `build.rs` invokes
`msgfmt`. To build OpenRig you need gettext installed:

| OS | Install |
|---|---|
| macOS | `brew install gettext` (Xcode CLT also ships with it) |
| Linux (Debian/Ubuntu) | `apt install gettext` |
| Linux (Arch) | `pacman -S gettext` |
| Windows | `choco install gettext` or `scoop install gettext` |

If `msgfmt` is missing, the build proceeds with a warning and the app
falls back to source-language passthrough — translations simply won't
load.

`slint-tr-extractor` is installed automatically by the extraction script
(`cargo install --locked slint-tr-extractor`) the first time you run it.

## Packaging

Each platform installer must place the compiled `.mo` files where the
runtime resolver looks for them:

- **macOS `.app`**:
  `OpenRig.app/Contents/Resources/translations/<lang>/LC_MESSAGES/adapter-gui.mo`
- **Linux `.deb`**:
  `/usr/share/openrig/translations/<lang>/LC_MESSAGES/adapter-gui.mo`
- **Windows installer**:
  `<install_dir>/translations/<lang>/LC_MESSAGES/adapter-gui.mo`

The build scripts under `scripts/` and `platform/` already pick up the
in-source `crates/adapter-gui/translations/<lang>/LC_MESSAGES/` directory
populated by `build.rs`.

## Switching language at runtime

Currently **requires app restart**. gettext only looks up `.mo` files
once per process, so live re-rendering would require either:

- Re-creating the `AppWindow` (loses runtime state), or
- A custom translation source plugged into Slint's compiler config

Tracked as a follow-up issue. The selector today saves the choice and
applies it on next launch.

## Translation guidelines

### Audio jargon stays in English

Music-production terminology that the global guitarist community reads in
English regardless of the UI language must **NOT** be translated. Leave
the `msgstr` empty (or repeat the source) in every catalog (pt-BR,
en-US, de-DE, fr-FR, …). gettext / rust-i18n then fall back to the source
key, which is already the right English term.

Examples — keep as-is in every language:

> Delay · Reverb · EQ · Tuner · Spectrum · Mute · Bypass · Gain · Send ·
> Return · Buffer · Sample Rate · Compressor · Limiter · Wah · Pitch ·
> Phaser · Flanger · Chorus · Overdrive · Distortion · Boost · Cab · IR ·
> NAM · LV2 · VST3

This convention reflects how the audio plugin world labels itself across
locales — translating these would actively confuse users instead of
helping them.

### Translate everything else

Section headers, button labels, dialog text, error messages, status
toasts — translate normally.

## Marking Rust strings

Use the `rust_i18n::t!` macro for any user-visible string emitted from
Rust (toast text, status messages, error strings shown in the UI):

```rust
use rust_i18n::t;
set_status_error(&w, &t, &t!("Projeto recente inválido."));
window.set_status_message(t!("Nenhum projeto carregado.").to_string().into());
return Err(anyhow!("{}", t!("Selecione pelo menos um input e um output antes de continuar.")));
```

After adding strings, add the corresponding entry to **every**
`crates/adapter-gui/locales/<lang>.yml`. The
`scripts/migrate-rust-strings.py --dry-run` script lists every key it
would migrate (use it to find strings that still need wrapping).

`rust_i18n::t!` returns `Cow<'static, str>` — call `.to_string()` when
crossing into Slint's `SharedString::from(...)` boundary.

## Out-of-scope (follow-up issues)

- **Conditional bindings** (`text: cond ? "a" : "b"` in `.slint`) — the
  migration script skips these since each branch needs per-context
  judgment.
- **`format!` calls embedding translatable strings** (e.g. `format!("Plugin error: {}", first.message)`) — manual migration to
  `t!("Plugin error: %{message}", message = first.message)` (rust-i18n
  placeholder syntax).
- **Right-to-left layouts** (Arabic, Hebrew) — Slint supports the layout
  direction property but no current locale uses it.
- **Translation of YAML content** (preset names, user-defined chain
  names) — these are user content, not UI chrome.
- **Audit other components against existing crates** — tracked in #359
  (see issue body for the framework-audit checklist).
