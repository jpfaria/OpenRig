# Per-chain multi-layer looper (#323) — Implementation Plan

> **For agentic workers:** implement task-by-task, RED first. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Ship a Boss RC-style multi-layer looper as a per-chain feature (like the DI loop): record the chain's dry input, play it back through the chain, overdub layers, undo/redo, speed/reverse/decay, N independent loopers per chain, MIDI footswitch control, panel UI, auto-save with the project.

**Architecture:** A `LooperBank` owned by one `ChainRuntimeState`. Layer buffers are allocated off the audio thread and handed over through a lock-free `ArrayQueue` (the `drain_pending_block_toggles` pattern, #580); playback sums the active layers on read, so undo/redo are O(1) and nothing allocates, locks or frees on the audio thread. Control flows GUI/MIDI/MCP → `Command` → `LocalDispatcher` → `Event` → runtime.

**Tech Stack:** Rust (engine/application/project/adapter-gui/adapter-midi), crossbeam `ArrayQueue`, Slint UI, hound for wav.

**Spec:** `docs/superpowers/specs/2026-07-23-issue-323-looper-design.md`

## Global Constraints

- Max loop length **60 s** stereo at engine sample rate; **8** layers per looper; **N** loopers per chain.
- Zero allocation / lock / syscall / I/O on the audio thread (invariant #8).
- Stream isolation: a bank belongs to exactly one runtime; nothing shared across chains (LEI ZERO).
- Never hardcode a sample rate — use the live rate (`controller.sample_rate()` / `engine_sr()`).
- Every state change is a `Command`; every value the GUI reads is a `QueryKind`.
- No `#[ignore]`; ≤600 LOC per `.rs`, ≤500 per `.slint`; zero warnings.
- Repo content in English; every new `@tr` key filled in all 9 locales in the same commit.
- TDD red-first: watch the assertion fail before writing production code.
